### 문제 1: QB 언더센터 위치 여러번 수정
- **증상**: QB y값을 -0.09 → -0.05 → -0.08로 세 번 수정
- **영향**: 사용자가 여러 번 피드백해야 했음
- **근본 원인 분석**:
  1. 왜 여러 번 수정했는가? → 사용자 요청 값을 정확히 이해 못함
  2. 왜 이해 못했는가? → 최초 요청 시 "겹치지 않게"만 언급, 구체적 값 없음
  3. 왜 구체적 값을 물어보지 않았는가? → 계획 단계에서 임의로 결정
  4. 왜 임의 결정했는가? → 사용자 선호를 확인하지 않음
  5. **근본 원인**: 시각적 요소는 사용자 확인 필요하다는 인식 부족
- **재발 방지책**: 위치/크기/색상 등 시각적 요소 변경 시 구체적 값 확인 후 진행
- **조치 상태**: [x] 완료

### 문제 2: Zone 블락 마커 오류
- **증상**: Zone 블락에 arrow 마커 사용 → 사용자가 t_block 요청
- **영향**: 사용자가 재수정 요청
- **근본 원인 분석**:
  1. 왜 arrow를 사용했는가? → 계획 문서에 "곡선 + 화살표"로 작성됨
  2. 왜 계획이 잘못됐는가? → 사용자 요청을 추측으로 해석
  3. 왜 추측했는가? → "Zone은 곡선"만 언급, 마커는 명시 안됨
  4. 왜 확인 안했는가? → 빠른 구현에 집중
  5. **근본 원인**: 불명확한 요구사항을 추측으로 채움
- **재발 방지책**: 요구사항 불명확 시 기존 동작 유지 또는 사용자 확인
- **조치 상태**: [x] 완료

### 문제 3: 라우트 트리 클릭 기능 누락
- **증상**: 라우트 트리 프리뷰만 구현, 클릭으로 라우트 생성 안됨
- **영향**: 사용자가 추가 기능 요청
- **근본 원인 분석**:
  1. 왜 클릭 기능이 없었는가? → 호버 프리뷰만 구현
  2. 왜 프리뷰만 했는가? → 계획에 "라우트 트리 표시"만 있었음
  3. 왜 상호작용을 고려 안했는가? → 사용자 의도 파악 부족
  4. 왜 의도 파악 못했는가? → "표시"가 단순 시각화로만 해석됨
  5. **근본 원인**: UI 기능의 완전한 사용 시나리오 미고려
- **재발 방지책**: UI 요소 추가 시 "보여주기만" vs "상호작용 가능" 명확히 확인
- **조치 상태**: [x] 완료

### 문제 4: 포트 충돌로 로컬 서버 확인 불가
- **증상**: "안나와" - 사용자가 포트 3000에서 확인, 실제 서버는 3003
- **영향**: 사용자가 변경사항 확인 못함
- **근본 원인 분석**:
  1. 왜 포트가 달랐는가? → 3000 포트가 이미 사용 중
  2. 왜 이전 프로세스가 남아있었는가? → 이전 세션에서 종료 안됨
  3. 왜 사용자에게 안내 안했는가? → 서버 시작 시 포트 변경 메시지 확인 안함
  5. **근본 원인**: 서버 시작 후 실제 포트 확인/안내 누락
- **재발 방지책**: dev 서버 시작 후 실제 포트 번호 사용자에게 명시
- **조치 상태**: [x] 완료
